### cpp基础

1. static_cast 在编译时做转化
2. static cast比c风格的cast要好 c风格的cast可以理解为在尝试各种各样的cast static cast条件更加严格更加安全
3. Curiously Recurring Template Pattern 奇异递归模板模式 把运行时多态转化成编译时 具体例子看runtime.cpp和crtp.cpp
4. 编译器编译主要经历四个阶段
   1 预处理阶段 展开一些头文件和#开头的语句 
   2 编译阶段 把语言转化成汇编码
   3 汇编阶段 把汇编码转化成机器码 其中会替换掉一些已经知道的地址 但是有许多地址还不知道
   4 链接阶段 把不同的汇编文件汇总起来 替换掉那些在汇编阶段不知道的地址 
5. buildin_expect() 和 [[likely]]的区别 
   buildin_expect限制一个语句的可能性 likely限制一个分支的可能性 
   if (buildin_expect) {} if() [[likely]] {} reopen
   行为上来讲都是对编译器的建议不一定被优化执行 
   通俗的理解cpu在取指令的时候会预先取likely修饰的指令来增快运行速度 
6. 左值和右值
   左值通俗的理解成可以放在表达式左边的值 有持续稳定可以被引用的身份 
   右值可以理解为表达式右边的值 只是一个值 相当于一个临时变量 写代码时 不能依赖他的身份和地址 
   引入左值和右值可以实现移动语义和完美转发 让编译器做更多的优化 
7. 指针减法像是同一个数组内的地址相减 返回的是元素地址之间的差 不同类型的指针之间的减法是非法的
8. 如果要直接拿信息的话可以recv(fd_, buf, sizeof(buf)) 但是通过recvmsg 并且使用msg可以控制更多的东西 可以通过msg_control拿到内核写的信息 通过msg_name拿到对端端口的信息 通过iov控制数据写到哪里
9. 在传一个tcp包的时候内核会加入一些control message 用CMSG_SPACE可以得到一段数据加上内核数据的真实大小
10. ambda一般情况下会比std::function要好 因为std::function内部可以装很多东西 比如普通函数 bind的结果 lambda 所以内部做了一些统一接口的操作 lambda的调用链更加简单 编译器更好做优化 
11. pack改变一个结构的align 可以用alignof获取一个结构的align 当没有pack的时候一个结构的align等于子结构align的最大值
12. (a -> b) (a -> c) (b -> d) (c -> d) ->表示在前面被后面include在头文件中 这个时候会出现redefintion问题 需要用pragma标记 他的作用类似于传统的#ifndef A_H #define A_H

1. 为什么需要内存池 
   在低延迟系统中 动态分配内存是很慢的 我们可以在启动的时候申请大量内存然后自定义使用的方式来达到更快的效果

### 交易系统生态

- 服务器层面   

  - market data publisher 市场数据发发布器用来将撮合引擎维护的限价订单簿的信息发送给所有市场参与者 一般用UDP多播的形式 在网络层做多播

  - matching engine 撮合引擎负责处理请求一般有添加修改需要订单等请求 还负责对价格交叉的订单进行匹配
  - order gateway server 订单服务网关 负责和市场交易者建立连接 接受他们发送的请求 TCP连接 
- 客户端层面
  - market data protocol decoder 市场数据消费者负责接收市场发布器发送的消息
  - order gateway protocol encoder & decoder 订单网关编码器和解码器客户端 负责和订单网关交互
  - trading engine 交易引擎 负责根据市场数据消费者的信息来构建一个和策略相关的定价簿进行分析之后做出自动决策
- 订单薄
  - 交易所把接收到的订单整合成一种数据结构 命名为order book 定价簿
  - 在订单到达撮合引擎的时候会先和被动订单进行匹配如果无法匹配就会变成被动订单
  - 被动订单按照价格排序 在买单中价格高的在先 在卖单中价格低的在先 相同价格的订单按照先进先出原则排序
	- 在修改订单的过程中如果订单数量减少在订单簿中的位置和优先级不变
	- 在修改价格或者数量增加的情况下相当于取消订单之后发送一个新订单
	- 匹配订单
- 设计撮合引擎
  - 服务器层面的各个部件要运行在不同的线程上 他们应该要独立工作 任何一部分的繁忙都不应该阻碍其他部分的运行
  - 撮合引擎需要和订单网关双向通信 订单网关告知引擎交易信息 缩合引擎提供交易信息响应和交易的处理情况
  - 撮合引擎需要单方面向市场数据发布器输送市场数据
- market data publisher 
  - market data protocol encoder 需要将撮合引擎的的市场信息进行编码
  - 由于使用UDP的多播所以交易者在接收信息的时候可以丢包所以需要定期向交易者发送快照 因此有一个单独的线程进行快照合成 
  - 会发布很多的信息到交易者 
	- 1. 市场状态比如关闭开盘正常交易时间段等 
	  2. 交易工具更新 比如最小价格增量 跳动单位等
	  3. 订单信息更新比如被动订单添加修改删除
	  4. 交易信息 交易工具id 交易价格和数量
	  5. 交易统计信息 开盘价 收盘价 最高价最低价等
- order gateway server 

  - tcp connection manager 和用户建立连接 处理不活跃的连接 
  - FIFO sequencer 保证公平性地接收数据再转发给撮合引擎 
  - order gateway protocol encoder & decoder 在撮合引擎格式和网关格式之间作转化
- market data consumer 

  - 订阅和消费交易所的多播网络流量 

  - 为了负载均衡 交易所会把不同的交易工具分组到不同的多播地址 

  - consumer要监听有用的交易工具和产品

  - 从交易所信息中解码

  - 因为UDP可能会丢包所以会有一些同步机制来保证获取的定价簿的状态

  - publisher会发布两种数据流 增量市场数据流和快照市场数据流 

    - 增量市场数据流要求的带宽小 在订阅这个的时候如果发生丢包就订阅快照数据流同时把增量数据流排队之后应用到快照上

    - 快照数据流要求带宽大发布频率低
- order gateway client 
	- encoder & decoder 负责把trading engine的格式和交易所请求格式之间完成转化 
	- tcp connection manager 负责和交易所建立可靠的tcp连接
- trading engine
	- 根据交易所发布的信息构建限价订单簿 有时并不需要构建完整的订单簿只需要关注需要的信息 这是构建的订单会和交易所中撮合引擎的订单有区别因为有各种延迟
	- 构建特征引擎 根据订单簿做分析
